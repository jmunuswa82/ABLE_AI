REQUIRED SYSTEM ARCHITECTURE
==================================================

Build or refactor the system into these layers:

1. INGESTION LAYER
- Upload `.als`
- Store file safely
- Create project record
- Compute artifact hash
- Persist metadata
- Track lifecycle state

2. ALS DECODER / PARSER
- Correctly load Ableton `.als`
- Handle compression/container format correctly
- Parse underlying XML safely
- Preserve unknown/unmapped nodes for round-trip safety

3. ROUND-TRIP XML AST
- Build a loss-minimizing XML tree representation
- Preserve node ordering
- Preserve attributes
- Preserve unknown sections
- Preserve untouched regions as much as possible

4. PROJECT GRAPH
Create a normalized internal model for:
- project metadata
- tempo
- arrangement length
- locators
- tracks
- return tracks
- master track
- clips
- devices
- automation envelopes
- routing/send relationships
- timeline positions
- IDs and references

5. ANALYSIS ENGINE
- Detect BPM accurately from actual Ableton source
- Detect if tempo automation exists
- Count tracks/clips/devices
- Identify arrangement sections
- Detect automation density and absence
- Detect transition weakness
- Detect intro/outro weakness
- Detect sidechain/routing opportunities
- Infer style profile from project characteristics
- Produce grounded, evidence-based recommendations

6. SUGGESTION ENGINE
Each suggestion must be:
- specific
- explainable
- grounded in evidence
- categorized
- prioritized
- confidence-scored
- marked exportable or manual-only

7. SELECTION / MUTATION PLANNER
- Allow user to choose suggestions
- Convert selected suggestions into typed mutation objects
- Reject unsupported unsafe mutations early

8. MUTATION ENGINE
- Apply typed mutations to ProjectGraph
- Never directly mutate raw XML from AI text
- Use centralized identity/reference management

9. ALS COMPILER
- Map mutated ProjectGraph back into XML AST
- Preserve untouched original structure where possible
- Insert new nodes in correct Ableton-compatible positions
- Maintain valid IDs and references
- Serialize valid `.als`

10. VALIDATION LAYER
Before export:
- XML well-formedness
- no duplicate IDs
- all references resolve
- required structure intact
- tempo source valid
- track/device/clip integrity valid
- mutation results actually present

11. EXPORT LAYER
- Store compiled artifact
- Mark export ready only when actual artifact exists
- Stream valid modified `.als` only

12. RESPONSIVE UI LAYER
- Build responsive layout logic as a first-class system requirement
- Support large desktop, laptop, tablet, and mobile breakpoints intentionally
- Reflow dense data panels into stacked, swipeable, collapsible, or segmented structures where appropriate
- Preserve critical actions and visibility on small screens without sacrificing clarity

AI SUGGESTION QUALITY REQUIREMENTS
==================================================

Suggestions must be advanced, specific, and musically useful.

No fluff like:
- “make it better”
- “improve transitions”
- “add more energy”

Instead generate suggestions like:
- Add 8-bar stripped intro from bars 1–9 for DJ-friendly entry
- Add low-pass automation on atmospheric return before breakdown at bars 61–65
- Add master utility gain fade over final 8 bars for controlled exit
- Add send automation burst on percussion bus into transition at bars 93–97
- Add arrangement locator for build section around bars 81–97
- Recommend sidechain compression between kick and bass due to sustained overlap
- Add outro locator and density reduction after bar 165
- Add filter envelope movement on texture layer due to static spectral profile
- Introduce contrast event before drop because density remains flat for 32+ bars

Each suggestion must include:
- suggestionId
- title
- category
- priority
- confidence
- reasoning
- evidence
- affected tracks
- target bars
- safeToExport boolean
- manualOnly boolean
- mutationType if exportable
