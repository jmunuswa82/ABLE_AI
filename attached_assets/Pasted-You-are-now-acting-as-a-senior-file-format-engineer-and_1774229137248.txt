PART 3 — ID GENERATION AND REFERENCE RULES
==================================================

This is mandatory and central.

Create a strict repo-level “ALS Identity Rules” guide and enforce it in code.

First inspect real ALS fixtures to understand how identity attributes and references behave.

Then implement these rules:

A. ID PRESERVATION
- existing IDs from source ALS must remain unchanged whenever possible

B. CENTRALIZED ALLOCATION
- all new IDs must come from one allocator only
- no scattered local ID generation
- no frontend-generated IDs
- no random ad hoc string concatenation

C. UNIQUENESS
- every newly created ID must be globally unique within the exported project

D. FORMAT SAFETY
- new IDs must match the numeric/string pattern actually observed in valid fixtures
- if project uses numeric IDs, do not invent UUID-style IDs
- if project uses a constrained pattern, honor that pattern

E. REFERENCE CONSISTENCY
- every reference to tracks/devices/parameters/envelopes must resolve
- if a new node creates a new target reference, the referenced target must exist
- if a node is cloned or inserted, all relevant references must be updated consistently

F. NO REUSE IN SAME COMPILE PASS
- deleted or replaced IDs must not be accidentally reused during the same export

Implement:
- scanExistingIds(ast)
- allocateNewId(kind)
- registerReference(source, target)
- validateReferenceClosure(ast/projectGraph)

Validation must detect:
- duplicate IDs
- malformed IDs
- missing referenced target
- broken envelope target references
- broken device parameter references
- orphan clip/device/track references
