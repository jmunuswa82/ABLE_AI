REQUIRED ARCHITECTURE
==================================================

Implement or refactor to this export architecture.

A. ALS INPUT LAYER
- Load `.als` safely (gzip-aware if needed)
- Parse XML preserving ordering and unmodified nodes wherever possible
- Preserve unknown tags and attributes for round-trip safety
- Support large projects efficiently

B. ROUND-TRIP XML AST
Create a round-trip-safe XML representation:
- tag name
- ordered attributes
- children order
- text values
- comments if present
- unknown nodes preserved
- exact subtree pass-through for untouched regions

Important:
Untouched sections must remain as close as possible to original source.
Do not regenerate the whole file from scratch if not necessary.
Prefer surgical subtree mutation with validated recomposition.

C. PROJECT GRAPH
Build an internal graph model for:
- project metadata
- tempo
- locators / arrangement sections
- tracks
- return tracks
- master track
- devices
- chains if relevant
- clips
- automation envelopes
- mixer/routing/send data
- muted/solo/arm states where relevant
- timeline/bar positions
- unique node identities
- parent-child and reference links

D. MUTATION ENGINE
AI modifications must be applied to the graph through typed operations only.

Examples:
- add MIDI clip
- add audio clip placeholder metadata
- add arrangement section locator
- add automation envelope points
- add send automation
- add Utility gain automation
- add filter sweeps
- create sidechain routing metadata if supported structurally
- update track names
- add generated tracks
- extend arrangement length

Do NOT mutate XML directly from AI output.
AI output must be transformed into typed graph mutations.

E. IDENTITY / REFERENCE MANAGER
Implement a dedicated ID manager.

This is mandatory.

The ID manager must:
- scan all existing IDs in the source ALS
- preserve existing IDs unchanged where possible
- generate new IDs only through a central allocator
- guarantee uniqueness
- ensure format validity for Ableton-compatible XML attribute usage
- update all corresponding references when new nodes are created
- never reuse deleted IDs in the same compile pass
- maintain deterministic behavior between runs

Build validation checks for:
- duplicate IDs
- null IDs
- malformed IDs
- orphan references
- broken parent references
- automation target references that do not exist
- device parameter references that do not exist

F. ALS COMPILER
Build a deterministic compiler that:
- applies graph changes back into the XML AST
- preserves original XML regions untouched where possible
- inserts new nodes in schema-consistent positions
- ensures required sibling ordering
- serializes valid XML only
- preserves stable formatting as much as practical
- writes a valid `.als` output file

G. PRE-EXPORT VALIDATOR
Before allowing download, run validations:
1. XML well-formedness
2. No duplicate IDs
3. All references resolve
4. Required root structure intact
5. Track/device/clip tree integrity
6. Automation envelope integrity
7. Bar/time values valid and non-negative
8. No illegal attribute encodings
9. No empty required elements
10. No accidental removal of critical project sections

If validation fails:
- do not export
- show exact failure reason
- show mutation step that caused failure
- write detailed debug log
